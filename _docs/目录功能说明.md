# HoCloth 目录功能说明

相关文档：
- [发布流程与构建说明](./发布流程与构建说明.md)
- [发布说明.md](./发布说明.md)
- [项目结构说明.md](./项目结构说明.md)

本文档说明当前仓库内各个主要目录和关键文件的职责，重点区分：
- 开发目录
- 运行目录
- 发布目录

## 顶层目录

### `.github/`

用于存放 GitHub Actions 工作流。
当前最重要的是 `workflows/release-addon.yml`，负责在云端调用 `package_addon.ps1` 打包插件并上传发布物。

### `_dist/`

用于存放本地打包输出的 zip 文件。
这是发布产物目录，不参与源码开发，也不应长期堆积旧包。

### `_docs/`

用于存放工程说明文档。
这里放的是开发文档，不会进入最终发布包。

### `_native/`

用于存放 C++ 子工程源码。
这是原生运行时的开发目录，主要包含：
- `CMakeLists.txt`
- `src/`
- `include/`
- 其他 native 构建配置

`_native/build/` 这类目录属于构建过程目录，不是 Blender 运行目录。

### `_third_party/`

用于存放第三方源码依赖。
当前主要包括：
- PhysX
- nanobind
- scikit-build-core

其中 PhysX 现在采用“源码目录内直接生成工程和编译产物”的方式：
- `_third_party/PhysX/physx/compiler/` 存放生成的 Visual Studio 工程
- `_third_party/PhysX/physx/bin/` 存放 PhysX 编译输出
- `_third_party/PhysX/packman-cache/` 存放 packman 缓存

这些内容只用于开发和编译，不进入最终发布包。

### `authoring/`

用于存放 Blender 侧的编辑、面板、操作器和 authoring 逻辑。
这是高频修改目录，所以不加 `_` 前缀。

### `components/`

用于存放 Blender 属性组件、组件注册和相关数据结构。

### `compile/`

用于把 authoring 数据编译成运行时可消费的数据结构。

### `runtime/`

用于存放 Python 侧运行时逻辑，以及与 native 模块之间的桥接代码。

### `_bin/`

这是 Blender 运行时真正使用的二进制目录。
应当只放插件运行所需内容，例如：
- `hocloth_native` 这类 Python 扩展模块
- native 运行时依赖的 DLL

不要把构建中间文件也放进 `_bin/`。

## 关键文件

### `__init__.py`

这是 Blender 识别插件的入口文件。
仓库根目录之所以可以直接作为 Blender 插件，就是因为这里有合法的 `__init__.py` 和 `bl_info`。

### `package_addon.ps1`

这是本地和云端共用的打包脚本。
它负责：
- 从仓库根目录收集插件运行需要的文件
- 排除开发目录和工程文件
- 生成 Blender 可安装的 zip

### `build_physx.ps1`

这是 PhysX 本地准备脚本。
它负责：
- 直接在 `_third_party/PhysX/physx` 内生成 PhysX 工程
- 使用 `_third_party/PhysX/packman-cache` 作为 packman 缓存
- 在可用时调用 `msbuild` 编译 PhysX

### `CMakeLists.txt`

这是仓库级 CMake 入口。
它的职责是协调 `_native/` 子工程，不属于插件运行时内容。

### `pyproject.toml`

这是工程级 Python 配置文件。
它属于开发配置，不进入发布包。

### `工程大纲.md`

这是项目蓝图文档。
它属于设计资料，保留在仓库中辅助开发，不进入发布包。

## 当前结构的核心原则

1. 仓库根目录本身就是 Blender 插件目录。
2. `_native/` 和 `_third_party/` 属于开发资源，`_bin/` 属于运行资源。
3. PhysX 在 `_third_party/PhysX/physx` 内直接构建，但这些生成物仍属于开发缓存，不参与发布。
4. 发布时依靠 `package_addon.ps1` 过滤开发内容，而不是手工删文件。
