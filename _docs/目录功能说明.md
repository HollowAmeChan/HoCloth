# HoCloth 目录功能说明

相关文档：

- [发布流程与构建说明](./发布流程与构建说明.md)
- [distribution.md](./distribution.md)
- [project_layout.md](./project_layout.md)

本文档专门说明当前仓库中每个顶层目录和关键文件的功能职责，方便后续扩展时不把“开发目录”“运行目录”“发布目录”混在一起。

## 顶层目录

### `.github/`

用于存放 GitHub Actions 工作流。

当前最重要的是：

- `workflows/release-addon.yml`

它负责在云端调用 `package_addon.ps1` 打包插件，并上传 artifact 或创建 GitHub Release。

### `_dist/`

用于存放本地打包输出的 zip 文件。

这个目录的定位是：

- 发布包输出目录
- 临时验证目录
- 不参与源码逻辑

建议把它当作一次性产物目录，定期清理旧包。

### `_docs/`

用于存放工程说明文档。

目前建议把这里的文档分成三类：

- 发布与构建流程
- 目录与结构说明
- 将来的设计说明或使用说明

### `_native/`

用于存放 C++ 子工程源码。

这个目录是“开发目录”，不是 Blender 运行目录。它可以包含：

- `CMakeLists.txt`
- `src/`
- `include/`
- `cmake/`

后续所有 native 构建中间文件，也应该优先落在 `_native/build/` 这类子目录里，而不是污染插件根目录。

当前 native CMake 已经约定：

- 运行时产物输出到插件根目录 `_bin/`
- 中间静态库和非运行时产物保留在构建目录内部

### `_third_party/`

用于存放第三方源码依赖。

这里现在主要放的是：

- PhysX
- nanobind
- scikit-build-core

它的作用是给 native 工程提供源码级依赖，不应进入最终发布包。

### `authoring/`

用于存放 Blender 侧入口逻辑、提取逻辑、操作器和面板。

这类目录会更频繁修改，所以不加 `_` 前缀。

### `components/`

用于存放 Blender 端组件系统，包括：

- 主组件目录
- 类型容器属性
- 组件注册表

### `compile/`

用于存放 authoring 数据到 compiled 中间数据的转换逻辑。

### `runtime/`

用于存放 Python 侧运行时会话和 native bridge。

## 关键文件

### `__init__.py`

这是 Blender 识别插件的入口文件，也是当前仓库最核心的运行文件。

它的职责是：

- 提供 `bl_info`
- 完成插件注册与反注册
- 管理插件根目录下的运行时路径，例如 `_bin/` 或 `_Lib/`

### `package_addon.ps1`

这是本地和云端共用的打包脚本。

它的职责是：

- 从仓库根目录收集插件运行需要的内容
- 排除开发态目录与工程文件
- 生成 Blender 可安装 zip

它不是构建脚本，而是发布整理脚本。

换句话说：

- `_native/` 负责编译
- `package_addon.ps1` 负责发布打包

### `CMakeLists.txt`

这是根级 CMake 入口文件。

它的定位是工程构建入口，不属于插件运行时内容。保留它是为了方便你从仓库根协调 `_native` 子工程。

### `pyproject.toml`

这是 Python 构建与工程元数据文件。

当前它更偏向工程配置文件，而不是 Blender 插件运行文件，所以发布时会被排除。

### `工程大纲.md`

这是项目蓝图文档，属于设计资料，不属于插件运行时内容。

它应该保留在仓库中帮助开发，但不进入发布 zip。

## 未来建议增加但当前可选的目录

### `_bin/`

建议作为 native 编译产物目录使用。

它的职责应该非常单纯：

- 存放 Blender 运行时所需的二进制文件
- 作为发布包中 native 运行时内容的唯一来源

不要把构建中间文件也放进 `_bin/`。

推荐理解为：

- `_native/build/` 是“构建过程目录”
- `_bin/` 是“最终运行目录”

当前 CMake 约定已经接好，后续只要增加真实 native target，运行时产物会默认落到这里。

## 当前结构的核心原则

现在这套目录设计的原则可以概括成三条：

1. 仓库根目录就是 Blender 插件目录
2. `_native/` 和 `_third_party/` 是开发资源，不是发布资源
3. 发布时靠 `package_addon.ps1` 过滤目录职责，而不是手工删文件

只要后续新增目录时也遵循这三条，整个仓库结构就会一直比较清晰。
