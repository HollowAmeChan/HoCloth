# 碰撞现状与边界说明

这个文档不再描述“统一架构草案”，而是说明当前碰撞系统的边界和方向。

## 当前主线

- 碰撞不再作为独立通用 XPBD contact 系统推进
- 碰撞被视为 BoneSpring 数据流中的一个局部阶段
- 重点是让碰撞投影配合骨长和方向约束工作

## 已经明确的边界

- 不继续以 `contact lambda` 为主线
- 不继续以 `speculative contact` 作为稳定性核心
- 不继续只用尾端点代表整根骨段
- 不继续让 `substep` 承担时间管理

## 现阶段应该保留的能力

- 骨段最近点检测
- 点对段 / 点对胶囊投影
- 碰撞后骨长整理
- fixed-step 驱动下的局部接触处理

## 经验结论

- 统一碰撞不是把所有东西塞进一个求解器
- 对 HoCloth 来说，更合理的是统一“状态流”和“处理顺序”
- 先统一数据语义，再统一碰撞实现
