# HoCloth 统一碰撞架构草案

这份文档用于结束当前“围绕 SpringBone 和静态碰撞体反复打补丁”的状态，给 HoCloth 后续的碰撞系统重构一个统一方向。

目标不是继续把当前 `apply_collider_constraints()` 调得更聪明，而是把碰撞正式提升为独立子系统，让下面这些需求走同一套底层：

- `SpringBone` 撞静态碰撞体
- `SpringBone` 撞会动的碰撞体
- 不同动态组之间的碰撞
- `ClothBone / MeshCloth / SpringBone` 共享碰撞世界
- 后续 `self collision`、摩擦、接触缓存、调试可视化

---

## 1. 为什么当前方向会越改越 hack

当前问题不在于“参数还没调对”，而在于系统边界不对。

现在 native 里的碰撞逻辑本质上还是：

1. SpringBone 自己持有碰撞引用
2. SpringBone 自己做最近点测试
3. SpringBone 自己在求解中间直接推位置
4. 再额外补速度、阻尼、反向 sweep 修正

这套方案做“骨链撞静态胶囊”还能跑，但一旦继续扩展，就会在结构上卡死：

- 会动的碰撞体需要速度和接触历史
- 两个动态组互撞需要双向约束，不是单向推出
- `ClothBone` 和 `MeshCloth` 不可能各自再写一套碰撞逻辑
- self collision 也不能靠 SpringBone 私有逻辑塞进去

所以现在最重要的结论是：

**“初始重叠自然贴开”这个思路是对的，但它应该成为统一碰撞世界里的接触约束行为，而不是 SpringBone 私有的补丁函数。**

## 2. 外部参考给出的统一结论

### 2.1 XPBD 提供的是统一约束框架，不是单独的 Spring 碰撞技巧

XPBD 的价值在于：

- 约束刚度对 `dt` 和迭代次数不那么敏感
- 距离、弯曲、接触都可以被视为统一约束

这意味着接触约束应该和 stretch / bend / attachment 一样，进入同一个 solver 流程，而不是事后手工补速度。

参考：

- XPBD, Macklin et al.  
  https://dl.acm.org/doi/10.1145/2994258.2994272

### 2.2 PositionBasedDynamics 给的是更接近 HoCloth 的工程结构

`PositionBasedDynamics` 开源库的启发不在于 API，而在于分层：

- model
- collision detection
- constraint set
- solver step

它支持多种接触对：

- point-point
- point-edge
- point-triangle
- edge-edge
- rigid contacts
- rods / Cosserat

这和 HoCloth 后面要同时支持 `SpringBone / ClothBone / MeshCloth` 的目标是同方向的。

参考：

- PositionBasedDynamics repository  
  https://github.com/InteractiveComputerGraphics/PositionBasedDynamics

### 2.3 刚性杆 / 骨链本质上更像 rod，而不是“尾点粒子”

当前 HoCloth 已经实测出一个事实：

- 只拿 bone tail 粒子做碰撞，结构信息不完整
- 补 tail tip 能改善，但仍然不是真正的 rod/segment 接触

后面如果要做稳定的骨链碰撞，方向应该是：

- chain element = segment / capsule
- contact primitive = segment-capsule、segment-plane、segment-segment

而不是继续围绕“尾点球”打补丁。

参考：

- Direct Position-Based Solver for Stiff Rods  
  https://diglib.eg.org/items/7e668ddc-ff48-4ad4-bdf6-167cc4a338ed

### 2.4 VRM / UniVRM 适合 authoring 层，不适合直接当 runtime 碰撞层

VRM 的 `Collider / ColliderGroup / Spring / Joint` 数据组织对 Blender 侧非常合适，说明 authoring 层应该继续保留这种思路：

- collider 独立
- collider group 独立
- spring 引用 group

但 runtime 层不能只停在这个层面，否则未来动态组互撞还是会继续 hack。

参考：

- VRM SpringBone docs  
  https://vrm.dev/en/univrm1/vrm1_tutorial/springbone/
- VRMC_springBone  
  https://vrm.dev/en/api/springbone/vrm1/

## 3. HoCloth 统一碰撞系统的目标分层

后续建议把 native runtime 明确拆成四层。

### 3.1 Model Layer

这里不再按产品名区分，而按 solver 需要的物理原语区分：

- `Particle`
- `Segment`
- `Triangle`
- `KinematicBody`
- `DynamicBody`
- `ColliderProxy`

对应到 HoCloth：

- `SpringBone` 主要产出 `Segment`
- `ClothBone` 主要产出 `Particle + Edge`
- `MeshCloth` 主要产出 `Particle + Triangle`

### 3.2 Collision World

统一维护所有碰撞对象，不属于某个 spring 私有。

最小版本建议支持：

- `Sphere`
- `Capsule`
- `Plane`

后续扩展：

- `SDF`
- `TriangleMeshProxy`

每个碰撞对象都有统一字段：

- `collision_object_id`
- `owner_component_id`
- `group_bits`
- `mask_bits`
- `motion_type`: `STATIC / KINEMATIC / DYNAMIC`
- `transform`
- `linear_velocity`
- `angular_velocity`
- `shape`

### 3.3 Contact Generation

从 CollisionWorld 中统一生成接触，不在 solver 内部临时找最近点。

第一阶段只做：

- `segment vs sphere`
- `segment vs capsule`
- `segment vs plane`
- `particle vs sphere/capsule/plane`

第二阶段再做：

- `segment vs segment`
- `particle vs triangle`
- `edge vs edge`
- `self collision`

输出统一 contact 结构：

- `feature_a`
- `feature_b`
- `normal`
- `penetration`
- `target_distance`
- `contact_point_a`
- `contact_point_b`
- `friction`
- `compliance`
- `cache_key`

### 3.4 Constraint Layer

把接触也正式纳入 XPBD 约束，而不是额外推力。

约束分类建议统一成：

- `DistanceConstraint`
- `BendConstraint`
- `AttachmentConstraint`
- `ShapeConstraint`
- `ContactConstraint`
- `FrictionConstraint`

这样“静态碰撞体”“动态碰撞体”“组间碰撞”只是 contact 的数据不同，不是逻辑分叉。

## 4. 对 HoCloth 当前代码的直接改造建议

下面不是终局设计，而是从当前代码出发最现实的迁移路线。

### 4.1 Python / compile 层继续保留 VRM 式 authoring

当前这部分方向基本是对的，应该保留：

- `COLLIDER`
- `COLLIDER_GROUP`
- `SPRING_BONE`
- 后续 `CLOTH_BONE`
- 后续 `MESH_CLOTH`

也就是说：

- authoring 继续使用 group 引用
- compile 继续把 Blender 对象烘平成稳定结构

但是 compile 输出需要从“只服务 SpringBone”升级为“统一服务 CollisionWorld”。

### 4.2 新增统一 compiled 结构

建议在 `compile/compiled.py` 里新增下面这些结构，而不是只继续扩 `CompiledCollider`。

```python
@dataclass
class CompiledCollisionObject:
    collision_object_id: str
    owner_component_id: str
    motion_type: str  # STATIC / KINEMATIC / DYNAMIC
    shape_type: str   # SPHERE / CAPSULE / PLANE
    world_translation: tuple[float, float, float]
    world_rotation: tuple[float, float, float, float]
    linear_velocity: tuple[float, float, float]
    angular_velocity: tuple[float, float, float]
    radius: float
    half_height: float
    group_bits: int
    mask_bits: int
```

```python
@dataclass
class CompiledCollisionBinding:
    component_id: str
    collision_filter_id: str
    collision_object_ids: list[str]
```

说明：

- `CompiledColliderGroup` 仍然保留，继续服务 authoring 和 UI
- 但 runtime build 时应该先把 group 解析成统一 collision binding

### 4.3 native 侧新增 `CollisionWorld`

建议先单独建模块，不继续往 `hocloth_runtime_api.cpp` 里堆。

推荐拆成：

- `_native/include/hocloth_collision_world.hpp`
- `_native/src/hocloth_collision_world.cpp`

最低职责：

- 持有 `CollisionObject`
- 更新 world transform
- broadphase
- narrowphase
- 输出 `ContactConstraintInput`

这样 `SpringBone solver` 不再直接遍历 collider list。

### 4.4 SpringBone 先升级为 segment-based contact，不再围绕 tail particle

当前代码已经说明：

- tail 粒子接触不够完整
- tail tip 只是几何补足，不是最终解

下一版的最低统一形式应该是：

- 每个 spring joint 输出一个 `segment`，从 parent/head 指向 tail
- contact 以 segment 为主，而不是粒子球为主

也就是说，后面 contact generation 输入应该更像：

```text
SegmentFeature {
  owner_component_id
  joint_index
  head_position
  tail_position
  radius
}
```

而不是现在的“某个 state.position + 某个 particle_radius”。

## 5. 动态组之间碰撞该怎么统一

这是这次讨论最关键的目标。

### 5.1 不再区分“静态碰撞逻辑”和“动态碰撞逻辑”

统一分类为三种对象：

- `STATIC`
- `KINEMATIC`
- `DYNAMIC`

含义：

- `STATIC`: 只更新 transform，不参与 solver 反作用
- `KINEMATIC`: transform 来自 Blender 或外部动画，速度由输入计算
- `DYNAMIC`: transform 由 solver 更新

然后所有碰撞都只走这套接触生成：

- spring vs static
- spring vs kinematic
- spring vs dynamic
- cloth vs dynamic
- dynamic vs dynamic

### 5.2 组过滤不要再只停留在“引用了哪个 collider group”

当前 `collider_group_ids` 适合 authoring，但不适合 runtime 扩展。

后续 runtime 层建议统一成 filter bits：

- `belongs_to`
- `collides_with`

这样以后你就可以表达：

- A 组只撞环境
- B 组撞环境和 A 组
- C 组只自碰不互碰

作者侧仍然可以保留 group UI，只是在 compile 里翻译成 bitmask。

### 5.3 接触缓存要预留

目前“震荡”“闪烁”“一碰上就反复摇摆”很大一部分来自：

- 每帧重新找最近点
- 没有接触记忆
- 没有稳定 manifold

所以统一 contact 结构里最好从第一版就预留：

- `cache_key`
- `previous_lambda`
- `lifetime`

哪怕第一阶段还不用完整 warm-start，也要把数据通道留出来。

## 6. 建议的分阶段重构顺序

### Phase A：先重构结构，不追求立刻物理更准

目标：

- 把当前碰撞从 SpringBone 私有逻辑里抽出来
- 统一变成 `CollisionWorld + ContactInput`

交付物：

- `CompiledCollisionObject`
- native `CollisionWorld`
- SpringBone 改成从 collision world 拉 contact

这一步做完，哪怕结果还是和现在差不多，也值得。

### Phase B：SpringBone 改为 segment/rod contact

目标：

- 不再围绕 tail particle
- spring bone 的碰撞原语统一成 segment/capsule

交付物：

- `SegmentFeature`
- `segment-capsule` / `segment-sphere` / `segment-plane`
- 可视化中直接显示 segment 接触

### Phase C：引入动态碰撞体和组间碰撞

目标：

- 支持 kinematic collider
- 支持 dynamic group 对 dynamic group

交付物：

- `motion_type`
- world velocity 输入
- 双向 contact constraints

### Phase D：扩展到 ClothBone / MeshCloth / self collision

目标：

- 用同一套 collision world 服务三条主线

交付物：

- `particle/triangle/edge` contact
- topology-aware self collision filter

## 7. 对当前实现的明确取舍

下面这几件事建议作为接下来一段时间的硬规则。

### 7.1 暂停继续打磨当前 `apply_collider_constraints()` 的行为细节

原因：

- 它的结构位置已经不对
- 继续调只会继续加 hack

### 7.2 保留当前确认有效的能力

这些不是问题，应该保留：

- 初始重叠自然贴开
- bone capsule 方向和可视化修正
- `Append Tail Tip`
- 视口半径/碰撞体可视化

### 7.3 下一步优先级不再是“调稳定性”，而是“拆系统”

优先级建议改成：

1. 统一 compiled collision data
2. native `CollisionWorld`
3. spring solver 改用统一 contact 输入
4. 再谈稳定性、摩擦、缓存、动组互撞

---

## 8. 对 HoCloth 的一句话路线建议

如果只保留一句路线判断，那就是：

**Blender 侧继续沿用 VRM 式 `Collider / Group / Spring / Joint` authoring，Native 侧改成 `CollisionWorld + XPBD Contact Constraints + Segment/Rod-based Spring Solver`。**

这条路线同时兼容：

- 你现在的 SpringBone
- 后面的 ClothBone
- 后面的 MeshCloth
- 以及未来“不同会动的组之间碰撞”

---

## 9. 下一轮建议直接落地的实现任务

如果按这个草案继续推进，下一轮最适合直接开始的不是调参数，而是下面这组结构任务：

1. 在 `compile/compiled.py` 新增统一 `CompiledCollisionObject`
2. 在 `compile/compiler.py` 把现有 collider/group 输出转成 runtime-ready collision list
3. 在 native 新建 `CollisionWorld` 模块，而不是继续往 `runtime_api.cpp` 堆
4. 给 SpringBone solver 改接口：输入改成 `contact list`，不再直接拿 collider list
5. 保留当前 behavior 作为临时 fallback，直到新的 contact 流程跑通

这五步做完之后，再回头处理震荡，才不会继续在错误边界上反复 patch。
