## 1. 当前构建模型

HoCloth 当前采用下面这套结构：

- 仓库根目录本身就是 Blender 插件目录
- `_native/` 保存 C++ 原生模块源码
- `_bin/` 保存 Blender 运行时真正加载的 native 产物
- `_third_party/` 只保留构建需要的第三方依赖
- 打包发布时排除 `_native/`、`_third_party/`、`_docs/` 等开发目录

## 2. 为什么这样调整

这次清理的目标是：

- 去掉旧版物理依赖的工程绑定
- 保留可以正常编译的 native 模块占位实现
- 让 Blender 侧 authoring、compile、runtime 流程继续可用
- 为后续自研 XPBD 解算器留出稳定接口

## 3. native 构建流程

### 3.1 推荐预设

当前推荐预设：

- `vs2022-release-native`
- `build-vs2022-release-native`

### 3.2 典型命令

```powershell
cmake -S . --preset vs2022-release-native
cmake --build --preset build-vs2022-release-native --config Release
```

构建成功后，`hocloth_native` 会输出到 `_bin/`。

## 4. 打包流程

本地打包脚本：

```powershell
powershell -ExecutionPolicy Bypass -File .\package_addon.ps1 -Version dev -IncludeNativeBuild -CreateZip
```

脚本会：

1. 可选构建 native 模块
2. 复制运行时必需文件到 staging 目录
3. 排除开发目录和文档
4. 生成 Blender 可安装 zip

## 5. 缓存与清理

当前主要需要关注的缓存只有两类：

### 5.1 native 构建缓存

- `_native/build/`
- `_native/cmake-build-*`

如果需要完整重编，直接清理这些目录即可。

### 5.2 打包缓存

- `_build/`
- `_dist/`

这些目录可以按需清理。

## 6. 当前最重要的结论

1. HoCloth 已经不再依赖旧版物理构建链。
2. native 模块现在是自包含的占位 XPBD backend。
3. Python 与 C++ 的桥接接口保持不变，方便后续替换为正式求解器。
4. 发布包只包含 Blender 运行所需内容，不包含 C++ 源码和开发依赖。
