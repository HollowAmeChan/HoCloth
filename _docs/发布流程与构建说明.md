# HoCloth 发布流程与构建说明

相关文档：

- [目录功能说明](./目录功能说明.md)
- [distribution.md](./distribution.md)
- [project_layout.md](./project_layout.md)

本文档说明 HoCloth 当前采用的发布模型、为什么这样组织目录，以及本地构建、云端发布、发布缓存管理分别应该怎么做。

## 1. 当前发布模型

HoCloth 采用的是：

- 仓库根目录本身就是 Blender 插件目录
- `_native/` 目录中放 C++ 子工程源码
- `_bin/` 目录中放编译后的运行时产物
- 发布 zip 时从仓库根目录打包
- 打包时排除 C++ 源码和开发辅助目录

这样做的好处是：

- 本地开发时，整个工程文件夹可以直接被 Blender 当成插件使用
- 不需要额外维护一份单独的“发布版插件目录”
- C++ 工程仍然可以和插件代码放在一个仓库里
- 发布包可以保持干净，不把 `_native/`、`_third_party/`、测试脚本等开发内容一起带出去

## 2. 目录职责

当前推荐按下面的职责理解目录：

- `__init__.py`
  Blender 插件入口。Blender 识别插件时最关键的就是这个文件。

- `_native/`
  C++ 子工程源码。这里放 CMake、头文件、源码、原生模块实现。

- `_bin/`
  编译完成后给 Blender 运行时实际使用的产物目录。
  这里未来应该放：
  - Python 扩展模块
  - 运行时 DLL
  - 必要的依赖动态库

- `_third_party/`
  第三方源码依赖，比如 PhysX、nanobind、scikit-build-core。

- `package_addon.ps1`
  本地打包脚本。它负责从仓库根目录收集插件文件，并排除不该进发布包的目录。

- `.github/workflows/release-addon.yml`
  GitHub Actions 发布工作流。它在云端执行打包脚本并上传发布包。

- `_dist/`
  本地打包输出目录。这个目录只放发布 zip，不参与源码管理。

## 3. 为什么根目录可以直接作为 Blender 插件

Blender 安装插件时，本质上关心的是：

- 插件目录里有没有 `__init__.py`
- 这个 `__init__.py` 里有没有合法的 `bl_info`
- 插件运行时能不能在自己的目录中找到所需 Python 模块和二进制依赖

只要这三点成立，插件目录里是否同时存在 `_native/`、`_third_party/`、`_docs/`，并不会妨碍 Blender 识别它。

所以当前这套结构的核心原则是：

- 开发态：允许目录里包含更多工程文件
- 发布态：通过打包脚本筛掉开发文件，只保留运行需要的内容

## 4. 手动本地构建流程

这里把“本地构建”分成两个层面：

- Python/插件层面：直接用当前目录做 Blender 插件
- C++/原生层面：把 `_native/` 编译出可运行产物，再拷到 `_bin/`

### 4.1 仅测试 Blender 插件入口

如果只是先验证插件能被 Blender 识别：

1. 确认仓库根目录有 `__init__.py`
2. 把整个 `HoCloth` 目录放到 Blender 的 `addons` 目录，或者用 Blender 的“从磁盘安装”选择发布 zip
3. 在 Blender 里启用插件

如果插件只依赖 Python 代码，而不依赖原生模块，那么到这一步就已经可以测试基础注册逻辑。

### 4.2 本地编译 C++ 子工程

未来 native 子工程建议遵循下面的输出约定：

1. 在 `_native/` 中配置 CMake 工程
2. 在 `_native/build/` 中执行本地构建
3. 把最终运行所需文件复制到仓库根目录 `_bin/`

推荐的目标是：

- 开发构建目录：`_native/build/`
- 安装或收集目录：`_bin/`

当前仓库已经把这条约定接进了 CMake：

- `CMAKE_RUNTIME_OUTPUT_DIRECTORY` 会指向插件根目录 `_bin/`
- `CMAKE_LIBRARY_OUTPUT_DIRECTORY` 会指向插件根目录 `_bin/`
- `CMAKE_ARCHIVE_OUTPUT_DIRECTORY` 会保留在构建目录下的中间产物目录

这意味着后续只要在 `_native/` 中增加真实的 `MODULE` 或 `SHARED` target，
它们的运行时产物默认就会落到 `_bin/`。

### 4.3 一个建议中的本地构建习惯

未来 native 工程接起来后，建议形成这样的操作顺序：

1. 在 `_native/` 里配置并编译
2. 把编译产物整理到 `_bin/`
3. 打开 Blender，直接加载当前仓库根目录作为插件
4. 如果要验证发布包，再运行一次打包脚本

这套顺序的核心思想是：

- `_bin/` 是插件运行时的真实依赖目录
- `_native/build/` 只是构建过程目录，不应该被 Blender 依赖

## 5. 本地打包发布

本地打包脚本是：

`package_addon.ps1`

当前使用方式：

```powershell
powershell -ExecutionPolicy Bypass -File .\package_addon.ps1 -Version dev
```

执行后会在 `_dist/` 中生成一个 zip。

### 5.1 打包脚本的工作原理

脚本不是直接把整个仓库压缩，而是：

1. 在临时目录中创建一个 `HoCloth/` 目录
2. 从仓库根目录复制要发布的内容进去
3. 排除不该发布的目录和文件
4. 把这个临时目录压缩为 zip

这样做的原因是：

- zip 内部需要是 `HoCloth/...` 这种插件目录结构
- 不能把 `.git`、`_native/`、`_third_party/` 之类的工程内容打进去
- 不能直接依赖“从仓库根压缩后再手工删文件”这种容易出错的流程

### 5.2 当前明确排除的内容

打包脚本当前会排除这类开发内容：

- `.git`
- `.github`
- `_build`
- `_dist`
- `_docs`
- `_third_party`
- `_native`
- `cpp`
- `__pycache__`
- 若干工程级文件，如 `.gitignore`

这意味着发布包默认只包含插件运行真正需要的内容。

### 5.3 `_bin/` 在打包中的角色

如果 `_bin/` 存在，并且里面放的是插件运行所需的二进制文件，那么它会跟随插件一起进入发布包。

这就是当前推荐把原生编译产物放进 `_bin/` 的根本原因：

- Blender 运行插件时就近加载
- 发布包天然自包含
- 不需要把整个 `_native/` 子工程一起发出去

## 6. 云端发布使用哪些文件

云端发布目前依赖两个核心文件：

- `.github/workflows/release-addon.yml`
- `package_addon.ps1`

职责分工是：

- GitHub Actions 工作流负责在云端触发流程
- PowerShell 脚本负责真正执行打包逻辑

### 6.1 GitHub Actions 触发方式

当前工作流支持两种触发方式：

- 推送 tag，例如 `v0.1.0`
- 手动触发 `workflow_dispatch`

### 6.2 云端发布原理

云端发布时，GitHub Actions 会：

1. checkout 仓库
2. 解析版本号
3. 调用 `package_addon.ps1`
4. 把 `_dist/` 中生成的 zip 上传成 artifact
5. 如果是 tag push，则把 zip 附加到 GitHub Release

也就是说，真正决定“什么会进发布包”的，不是 workflow 本身，而是打包脚本。

这个设计有一个很实际的好处：

- 本地打包和云端打包用的是同一套规则

这样你本地验证通过的打包结果，云端通常也会保持一致。

## 7. 如何手动管理发布版缓存

这里的“发布版缓存”可以分成三类看：

- 本地打包缓存
- 本地构建缓存
- GitHub Release 成果物

### 7.1 本地打包缓存

本地打包缓存主要是：

- `_dist/` 中旧 zip
- `_build/` 中旧的打包临时目录

这些文件都不应该长期保留太多。

推荐做法是：

- 每次准备正式发布前，先清空 `_dist/`
- 如果 `_build/` 里有旧的 `package-*` 目录，也一起清理

### 7.2 本地构建缓存

本地构建缓存主要是：

- `_native/build/`
- 未来可能的 `cpp/build/`
- 编译过程中生成的中间文件

这些目录是开发缓存，不是运行时资产。

建议规则是：

- 想彻底重编时，删掉 `_native/build/`
- 真正需要被插件使用的，只保留到 `_bin/`
- 不要把 `_native/build/` 当作发布目录

### 7.3 发布版二进制缓存

如果 `_bin/` 中保存的是最近一次构建的产物，那么它实际上也是一种“运行时缓存”。

你可以按下面方式管理：

- 如果要确保插件始终用最新 native 版本，重新构建后覆盖 `_bin/`
- 如果要回退到旧版 native，先备份旧 `_bin/`
- 如果某次构建明显有问题，优先清空 `_bin/` 后再重新复制一遍产物

### 7.4 GitHub 发布缓存

GitHub 端常见的“缓存”主要是：

- workflow 产生的 artifact
- Release 页面上挂着的历史 zip

手动管理方式：

- 不再需要的 artifact 可以在 Actions 页面删除
- 不再需要的旧 Release 包可以在 Releases 页面删除或替换
- 如果某个 tag 发布错了，应该修正版本号并重新发布，而不是默默覆盖同名版本

## 8. 推荐的实际工作流

结合当前项目状态，比较稳的工作流是：

1. 平时开发直接在仓库根目录做 Blender 插件开发
2. C++ 代码在 `_native/` 中开发和编译
3. 编译完成后把运行时文件整理到 `_bin/`
4. 本地先跑一次 `package_addon.ps1`
5. 用生成的 zip 在 Blender 中做安装验证
6. 验证没问题后，推送 tag 触发 GitHub Release

## 9. 当前需要记住的结论

最重要的结论就三条：

1. 仓库根目录就是 Blender 插件目录
2. `_native/` 是源码工程，`_bin/` 是运行时产物目录
3. 发布时靠 `package_addon.ps1` 排除开发内容，而不是手工删文件
