# HoCloth 发布流程与构建说明

相关文档：

- [distribution.md](./distribution.md)

本文档说明 HoCloth 当前采用的开发、构建和发布流程，以及各个目录在这个流程中的角色。

## 1. 当前发布模型

HoCloth 当前采用的是：
- 仓库根目录本身就是 Blender 插件目录
- `_native/` 放 C++ 子工程源码
- `_bin/` 放 Blender 运行时真正加载的二进制产物
- PhysX 直接在 `_third_party/PhysX/physx` 内生成工程和编译输出
- 打包发布时排除 `_native/`、`_third_party/`、`_docs/` 等开发目录

这样做的好处是：
- 本地开发时，整个工程目录可以直接被 Blender 当作插件使用
- native 编译产物统一落到 `_bin/`，运行关系清晰
- PhysX 路径最直接，不需要额外同步一份源码工作目录
- 发布包干净，不会把 C++ 源码和第三方开发依赖一起打出去

## 2. 目录职责

当前最关键的目录分工如下：

- `__init__.py`
  Blender 插件入口。
- `_native/`
  native 源码和 CMake 工程。
- `_bin/`
  插件运行时真正使用的本地模块和 DLL。
- `_third_party/`
  第三方源码依赖，比如 PhysX、nanobind、scikit-build-core。
- `_dist/`
  本地打包输出目录，只放发布 zip。

## 3. 为什么根目录可以直接作为 Blender 插件

Blender 识别插件时，核心只关心：
- 目录中是否有 `__init__.py`
- `__init__.py` 是否提供合法的 `bl_info`
- 运行时能否从插件目录找到所需 Python 模块和二进制依赖

因此，仓库里即使同时存在 `_native/`、`_third_party/`、`_docs/` 等开发目录，也不会妨碍 Blender 把根目录识别为插件。

## 4. 本地构建流程

### 4.1 Python / Blender 插件层

如果只是验证插件入口是否正常，直接把当前 `HoCloth` 目录放进 Blender 的 `addons` 目录即可。

### 4.2 PhysX 准备

执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\build_physx.ps1
```

现在这个脚本会直接在 `_third_party/PhysX/physx` 里工作。

最重要的几个位置是：
- `_third_party/PhysX/physx/compiler/...`
  这是 PhysX 生成出来的 Visual Studio 工程目录
- `_third_party/PhysX/physx/bin/win.x86_64.../release`
  这里才是实际的 PhysX `.lib` 和 `.dll` 输出目录
- `_third_party/PhysX/packman-cache/`
  这里是 packman 依赖缓存

### 4.3 native 编译

HoCloth 的 native 工程放在 `_native/` 下编译，但最终运行时产物会输出到 `_bin/`。

也就是说：
- `_native/build/` 是构建过程目录
- `_bin/` 是 Blender 运行时目录

### 4.4 自动链接 PhysX

当前 native CMake 默认会优先从 `_third_party/PhysX/physx/bin/win.x86_64.../release` 自动探测 PhysX 库文件。
如果该目录下存在标准 PhysX 输出，HoCloth 会自动链接这些库，并在构建 `hocloth_native` 后把所需 DLL 复制到 `_bin/`。

## 5. 本地打包发布

本地打包脚本是：

```powershell
powershell -ExecutionPolicy Bypass -File .\package_addon.ps1 -Version dev
```

执行后会在 `_dist/` 中生成 zip。

打包脚本不是直接压整个仓库，而是先构建一个临时发布目录，再排除开发内容后压缩。

## 6. 打包时会排除什么

当前明确不进入发布包的主要内容包括：
- `.git`
- `.github`
- `_build`
- `_dist`
- `_docs`
- `_native`
- `_third_party`
- 各类本地构建缓存和工程配置文件

这意味着发布包默认只包含插件运行真正需要的文件。

## 7. 云端发布

云端发布依赖两部分：
- `.github/workflows/release-addon.yml`
- `package_addon.ps1`

职责分工是：
- GitHub Actions 负责触发流程和上传产物
- `package_addon.ps1` 负责真正决定哪些文件进包

## 8. 缓存管理建议

本地需要关注的缓存主要有三类：

### 8.1 打包缓存

- `_dist/`
- `_build/`

这两类可以定期清理，尤其是在准备正式发布前。

### 8.2 native 构建缓存

- `_native/build/`

如果需要彻底重编，可以删除整个构建目录后重新配置。

### 8.3 PhysX 构建缓存

- `_third_party/PhysX/packman-cache/`
- `_third_party/PhysX/physx/compiler/`
- `_third_party/PhysX/physx/bin/`

这些都属于本地开发缓存，不应提交到仓库。
如果 PhysX 构建状态混乱，优先清理这些目录后重新执行 `build_physx.ps1`。

## 9. 当前最重要的结论

1. 仓库根目录本身就是 Blender 插件目录。
2. `_native/` 负责开发，`_bin/` 负责运行。
3. PhysX 直接在 `_third_party/PhysX/physx` 里构建，不再额外复制一份工作目录。
4. 发布时依靠 `package_addon.ps1` 排除开发内容，而不是手工删文件。
