## HoCloth 开发蓝图

### 1. 项目定位与目标

- **项目名称**：HoCloth
- **核心理念**：结合 **Magica Cloth 2 (MC2)** 的极致易用性与 **Havok** 的工业级稳定性上限。
- **目标用户**：
    - Blender 用户，侧重离线动画预览与快速迭代
    - 游戏美术开发者，侧重进引擎的骨骼物理逻辑与可控导出
- **差异化竞争**：
    - **对比 MMDTools / BoneX**：彻底摒弃 Blender 原生刚体约束系统，转而使用 PhysX 5 的 **PBD/XPBD** 与 **Reduced Coordinate Articulations**，解决大量约束下的性能下降与物理抖动问题。
    - **工作流优化**：从“手动连刚体参数”转向“组件化列表 + 笔刷权重驱动”的作者工作流。
    - **架构取向明确**：Blender 负责 authoring、UI、数据持久化与场景提取，C++ 负责构建运行时与解算，结果再同步回 Blender。

---

### 2. 技术栈选型

- **界面与逻辑层（Python / Blender）**
    
    - 负责插件 UI
    - 负责 authoring 数据持久化
    - 负责从 Blender 场景提取骨骼、碰撞体、代理网格、权重等输入
    - 负责把运行结果同步到 pose / cache 工作流
- **解算核心层（C++）**
    
    - 使用偏 C 风格的 C++ 封装运行时逻辑
    - 不向 Python 暴露 PhysX 原生对象生命周期
    - 所有运行时对象通过 handle 管理
- **通信层（nanobind + scikit-build-core + CMake）**
    
    - Python 与 C++ 之间只交换紧凑、连续的 buffer
    - 优先使用 numpy.ndarray 传输骨骼变换、粒子位置、碰撞参数、拓扑索引等数据
    - 高频路径避免 Python 对象级循环调用
- **物理内核（PhysX 5.3+）**
    
    - PxArticulationReducedCoordinate：骨骼链、高稳定性、不拉伸
    - PxParticleSystem：PBD 布料、代理网格、后续软体扩展
    - 自定义过滤逻辑 / Spatial Hashing：用于复杂碰撞排除与命名碰撞层

---

### 3. 系统总体原则

- **结构性修改不做实时同步**
    
    - 骨骼链结构、碰撞体结构、代理网格拓扑、组件增删改等，统一通过 Rebuild 生效
- **运行时只接受紧凑输入**
    
    - 运行时每帧只接收 root / animated target / 外力 / dt 等高频输入
    - 不在每帧重新发送结构性配置
- **骨骼实时，网格缓存**
    
    - 骨骼结果优先实时回写
    - 顶点级网格结果优先输出为缓存文件，不承诺高模实时回写
- **Blender 只保存 authoring 数据**
    
    - 不在 Blender 自定义属性里保存 C++ 运行时状态
    - C++ scene/runtime 状态只通过 handle 存活于运行时
- **Python 与 C++ 之间不传复杂宿主对象**
    
    - 传递的应当是 buffer、枚举、轻量配置、句柄，而不是 Blender/PhysX 对象引用

### 4. 核心架构设计

#### 4.1 分层结构

整体拆成三层：

- **Authoring / UI 层**
    - Blender 面板、组件列表、笔刷权重入口、调试显示入口
- **Build / Compile 层**
    - 从 Blender 场景提取数据并编译成 C++ 运行时 scene
- **Runtime / Cache 层**
    - 运行 PhysX
    - 返回骨骼变换
    - 输出代理网格缓存文件

#### 4.2 C++ 模块职责

建议明确拆分为三个模块：

- **Builder / Compiler**
    
    - 接收 Blender 提取的结构化输入
    - 构建 articulation / particle / collider 等 runtime scene
    - 输出 scene handle
- **Runtime / Scene**
    
    - 维护 scene 生命周期
    - 设置高频输入
    - step 模拟
    - 查询骨骼变换 / 粒子结果 / debug 信息
- **Exporter / Cache**
    
    - 负责把代理网格或布料结果写成 pc2
    - 未来扩展 abc

这三层的边界要尽量硬，不要混成“一个大模块什么都做”。

---

### 5. Blender 端组件系统设计

#### 5.1 组件组织方式

采用“双层容器”方案：

- 一个主 CollectionProperty 作为**组件目录**
    
    - 负责顺序
    - 负责启用状态
    - 负责 UI 展示
    - 负责组件类型分派
- 每种组件类型一个单独的 CollectionProperty 作为**类型容器**
    
    - 不直接显示在 UI
    - 专门存对应 PropertyGroup
    - 作为真实参数数据库

例如：

- hocloth_components
- hocloth_bone_chain_components
- hocloth_collider_components
- hocloth_proxy_mesh_components
- hocloth_cache_output_components
- hocloth_grab_trigger_components
- hocloth_softbody_components

#### 5.2 主组件项的职责

主组件项只存“身份和排序信息”，不存具体物理参数。建议字段：

- component_id
- component_type
- container_index
- display_name
- enabled
- ui_expanded

这里最重要的是：

- **component_id 是稳定主键**
- **container_index 只是缓存索引**
- 长期关系不要只依赖 index

#### 5.3 类型容器项的职责

每个具体组件项保存：

- component_id
- 该类型自己的 authoring 参数

例如 Collider 组件容器项里保存：

- component_id
- radius
- height
- direction
- offset
- collision_layer_name
- 其他碰撞参数

#### 5.4 同步原则

不把“刷新同步”作为常规数据流，而是作为修复与重建机制。

正常操作应当是：

- 新建组件时，同时写主列表和类型容器
- 删除组件时，同时删主列表和类型容器
- 复制组件时，同时复制两边
- 移动顺序时，只改主列表顺序
- refresh/sync 只用于重建 container_index 或修复异常引用

这样可以降低 undo/redo、递归 update、列表错位等问题。

#### 5.5 组件注册表

建议建立统一 registry，而不是在 UI / operator 里手写大量 if type == ... 分派。

例如 registry 至少负责：

- 组件类型名
- 对应容器属性名
- 对应 PropertyGroup 类型
- 创建函数
- 删除函数
- 查询函数
- 从 component_id 重建索引的函数

这层会直接决定后续组件系统是否可维护。

### 6. 数据模型与编译态设计

#### 6.1 Authoring 数据

Blender 侧保存的是作者数据，例如：

- 骨链选取
- 组件启用状态
- 碰撞体半径/高度
- 参数曲线
- 代理网格绑定关系
- 权重源
- 缓存输出设置

#### 6.2 Compiled 数据

送给 C++ 之前，要先编译为扁平、稳定的中间结构，例如：

- bone hierarchy
- rest pose
- parent indices
- collider descriptors
- articulation joint settings
- particle topology
- proxy mapping data
- cache descriptor

也就是说：

- UI 参数不是运行时参数
- Blender 属性不是直接喂给 PhysX
- 中间必须有一层“编译输出格式”

这层非常重要，后面会大大减轻维护压力。

### 7. IO 与性能策略

#### 7.1 高频路径优化原则

高频路径只传：

- 骨骼 root / target 变换
- 少量 runtime 参数
- dt
- step 次数
- 返回骨骼变换或粒子位置

避免：

- 每帧重建 scene
- 每帧传大批 Python 对象
- 每帧做 authoring 层解析
- 每帧重新查骨骼名、对象名、层级关系

#### 7.2 Blender 端真正的瓶颈判断

真正危险的通常不是 Python <-> C++ 边界，而是 Blender 回写与依赖图更新。

因此策略是：

- **骨骼结果实时回写**
- **网格结果尽量不走实时高模更新**
- **高模网格优先通过缓存文件驱动**

#### 7.3 Mesh 代理与缓存方案

顶点级解算结果不做高频实时传输，而是：

- 每个布料 mesh 拥有一个缓存文件
- C++ 输出 pc2
- Blender 使用 MeshCache 修改器读取
- abc 作为未来扩展支持

这样能有效规避“每帧大量顶点从 Python 回写 Blender mesh”的瓶颈。

---

### 8. 骨骼链与网格两条产品线

#### 8.1 骨骼链物理

这是 MVP 和第一阶段主线，目标是：

- 可实时调试
- 高稳定
- 适合尾巴、裙摆骨、挂件骨、发辫骨、胸部骨等骨骼物理
- 可服务于游戏引擎工作流

#### 8.2 代理网格 / 布料

这是第二主线，但不作为第一阶段核心承诺。主路线是：

- 低模代理网格解算
- 映射回高模
- 输出到缓存
- Blender 通过 MeshCache 播放结果

这条线重点放在离线动画预览和可缓存结果，而不是追求第一时间做成高模实时视口同步。

---

### 9. C++ Runtime API 方向

建议 API 形状尽量保持窄接口、句柄化、buffer 化。

例如逻辑上分成：

- build_scene(...) -> handle
- destroy_scene(handle)
- set_runtime_inputs(handle, ...)
- step_scene(handle, dt, substeps)
- get_bone_transforms(handle, out_buffer)
- export_proxy_cache(handle, path, frame_range, sample_rate)

原则：

- 结构数据只在 build 阶段进入
- 运行时不暴露 PhysX 对象
- Python 只看到 handle、数组、轻量配置

---

### 10. 缓存系统设计

建议从一开始定义 SimulationCacheDescriptor 概念，用于判断缓存是否有效。至少应包含：

- source object id
- topology hash
- frame range
- sample rate
- solver settings hash
- cache format
- cache path

这样后续可以可靠判断：

- 拓扑变了是否失效
- 参数变了是否失效
- 帧范围变了是否需要重烘焙
- pc2 与未来 abc 是否可共用同一套描述逻辑

### 11. 路线规划

#### 第一阶段：MVP 宿主架构验证

目标是先验证整条宿主链路，不碰大而复杂的 authoring 自动化。

- Blender 读取一条骨链
- 一键 build 成 C++ runtime scene
- C++ 保留 scene handle
- 每帧输入 root / animated target
- C++ step
- 返回每节 bone transform
- Blender 同步 pose
- 支持 reset / rebuild / destroy
- 明确 build 与 runtime 的分层边界

这个阶段的关键不是功能多，而是证明：

- Blender 数据提取方式可行
- nanobind 数据交换可行
- PhysX articulation 路线可行
- pose 回写速度可接受

#### 第二阶段：组件系统与工作流构建

在骨链 MVP 稳定后，开始建立正式 authoring 工作流。

- 落地主组件列表 + 各类型容器的双层结构
- 建立组件 registry
- 建立 component_id 主键机制
- 实现组件新建、删除、复制、重排
- 开发“物理组管理器”
- 实现一键将骨骼链转为 PhysX 解算对象
- 集成碰撞体自动化
- 在骨骼链上工作，使用默认半径、高度和参数曲线批量生成胶囊体
- 引入 PVD (PhysX Visual Debugger) 支持
- 适当加入 Blender 3D 视口调试辅助

这里的重点是“好不好用”，不是先追求功能上限。

#### 第三阶段：性能评估与功能上限突破

先做性能剖析，再决定是否推进 GPU 路线。

- 评估 IO 是否成为瓶颈
- 评估 Blender pose 回写成本
- 评估 mesh cache 导出链路成本
- 如果 IO 不是核心问题，再实现 CUDA / GPU 加速开关
- 开发自定义过滤器（Bitmask / 命名碰撞层）
- 支持用组件层名称进行碰撞过滤
- 开始支持 PBD 相关内容
- 实现代理网格蒙皮映射算法
- 输出到 pc2，Blender 读取缓存
- 未来再扩展 abc
- 视需要加入更完整的 debug / profiling 面板

#### 第四阶段：扩展与实验功能

这些功能保留为扩展路线，不提前污染 MVP 主架构。

- 实现抓取触发器
    - 允许用户用 Blender 自定义数据切换抓取状态
    - 支持布料拖拽或局部交互
- 实现软体类型
    - 支持可拉伸的物理效果
    - 作为 PBD/XPBD 扩展路线
- 深化 3D region GPU 可视化交互
    - 参考 BoneX 的交互逻辑
    - 让用户在视口中直接对照调节配置

这里建议明确标注为：

- Production Core：骨链、碰撞、缓存输出
- Experimental：Grab Trigger、Soft Body、高级实时交互

### 13. 当前最优先事项

如果按可执行性排序，当前最值得先做的是：

1. 建立 nanobind + scikit-build-core + CMake + PhysX 的最小工程
2. 打通骨链 build -> step -> pose 回写 闭环
3. 把 Blender 端组件系统骨架先落地成“主列表 + 类型容器 + registry”
4. 建立 authoring 数据到 compiled 数据的导出层
5. 等骨链闭环稳定后，再推进碰撞自动化与 PVD
6. 最后再进入代理网格缓存路线

---

### 14. 总结

HoCloth 的核心路线已经很明确：

- **实时主线**：骨骼链物理
- **离线主线**：代理网格 / 布料缓存
- **架构主线**：Blender 做 authoring，C++ 做 runtime，二者通过 buffer 和 handle 通信
- **工程主线**：先证明宿主架构，再做工作流，再做性能上限
