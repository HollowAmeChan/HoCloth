## HoCloth 开发蓝图

### 1. 项目定位与目标

- **项目名称**：HoCloth
- **核心理念**：模仿 Magica Cloth 2 这类“作者体验优先”的效果和工作流，但底层走自研 XPBD 路线，而不是继续绑定大型第三方物理 SDK。
- **目标用户**：
  - Blender 用户，侧重离线动画预览与快速迭代
  - 游戏美术开发者，侧重可控的骨链物理、代理网格与缓存导出
- **差异化方向**：
  - 不依赖 Blender 原生刚体系统
  - 不依赖外部通用物理 SDK
  - 由 Blender 负责 authoring、UI、数据持久化与场景提取
  - 由 C++ 负责 runtime 构建与 XPBD 解算

---

### 2. 技术栈选型

- **界面与逻辑层（Python / Blender）**
  - 插件 UI
  - authoring 数据持久化
  - 从 Blender 场景提取骨链、碰撞体、代理网格、权重等输入
  - 把运行结果回写到 pose 或缓存
- **解算核心层（C++）**
  - 使用偏 C 风格的 C++ 封装 runtime
  - 不向 Python 暴露底层求解器内部对象生命周期
  - 所有运行时对象通过 handle 管理
- **通信层（nanobind + CMake）**
  - Python 与 C++ 之间只交换稳定的 buffer、轻量配置和句柄
  - 高频路径避免 Python 对象级循环调用
- **物理内核（自研 XPBD）**
  - 骨骼链距离约束
  - 弯曲约束
  - 形状保持约束
  - 后续扩展到纯粒子布料和代理网格

---

### 3. 系统总体原则

- **结构性修改不做实时同步**
  - 骨链结构、碰撞体结构、代理网格拓扑、组件增删改统一通过 `Rebuild` 生效
- **运行时只接受轻量输入**
  - 每帧只接收 root、动画目标、外力、`dt` 等高频输入
- **Blender 只保存 authoring 数据**
  - 不在 Blender 自定义属性里保存 runtime 内部状态
- **Python 与 C++ 之间不传复杂宿主对象**
  - 只传 buffer、枚举、轻量配置、句柄

---

### 4. 核心架构设计

#### 4.1 分层结构

- **Authoring / UI 层**
  - Blender 面板、组件列表、笔刷权重入口、调试入口
- **Build / Compile 层**
  - 从 Blender 场景提取数据并编译成稳定的 runtime scene
- **Runtime / Cache 层**
  - 运行 XPBD 解算
  - 返回骨骼变换
  - 输出代理网格缓存文件

#### 4.2 C++ 模块职责

- **Builder / Compiler**
  - 接收 Blender 提取后的结构化输入
  - 构建 runtime scene
  - 输出 scene handle
- **Runtime / Scene**
  - 维护 scene 生命周期
  - 设置高频输入
  - 执行 `step`
  - 查询骨骼变换、粒子结果和调试信息
- **Exporter / Cache**
  - 输出 `pc2`
  - 后续扩展到 `abc`

---

### 5. Blender 端组件系统设计

#### 5.1 组件组织方式

采用“双层容器”方案：

- 一个主 `CollectionProperty` 作为组件目录
- 每种组件类型一个独立的 `CollectionProperty` 作为类型容器

例如：

- `hocloth_components`
- `hocloth_bone_chain_components`
- `hocloth_collider_components`
- `hocloth_proxy_mesh_components`
- `hocloth_cache_output_components`

#### 5.2 主组件项的职责

主组件项只保存身份和排序信息，不直接保存具体物理参数。

建议字段：

- `component_id`
- `component_type`
- `container_index`
- `display_name`
- `enabled`
- `ui_expanded`

---

### 6. 数据模型与编译态设计

#### 6.1 Authoring 数据

Blender 侧保存的是作者数据，例如：

- 骨链选取
- 碰撞体半径与高度
- 参数曲线
- 代理网格绑定关系
- 权重源
- 缓存输出设置

#### 6.2 Compiled 数据

送给 C++ 之前，先编译成稳定的中间结构，例如：

- bone hierarchy
- rest pose
- parent indices
- collider descriptors
- particle topology
- proxy mapping data
- cache descriptor

原则：

- UI 参数不是运行时参数
- Blender 属性不是直接喂给求解器
- 中间层必须稳定、可导出、可调试

---

### 7. IO 与性能策略

#### 7.1 高频路径优化原则

高频路径只传：

- root / target 变换
- 少量 runtime 参数
- `dt`
- `substeps`
- 骨骼变换或粒子位置结果

避免：

- 每帧重建 scene
- 每帧传大批 Python 对象
- 每帧重复解析 authoring 层

#### 7.2 Blender 端真正的瓶颈判断

高风险瓶颈通常不是 Python <-> C++ 边界，而是 Blender 的回写和依赖图更新。

因此策略是：

- 骨骼结果实时回写
- 网格结果尽量不做高频实时高模更新
- 高模网格优先通过缓存驱动

---

### 8. 产品主线

#### 8.1 骨链物理

这是 MVP 和第一阶段主线，目标是：

- 可实时调试
- 高稳定
- 适合尾巴、裙摆骨、挂件骨、发辫骨等骨链物理

#### 8.2 代理网格 / 布料

这是第二条主线，重点是：

- 低模代理网格解算
- 映射回高模
- 输出到缓存
- Blender 通过 MeshCache 播放结果

---

### 9. C++ Runtime API 方向

建议 API 保持窄接口、句柄化、buffer 化：

- `build_scene(...) -> handle`
- `destroy_scene(handle)`
- `set_runtime_inputs(handle, ...)`
- `step_scene(handle, dt, substeps)`
- `get_bone_transforms(handle, out_buffer)`
- `export_proxy_cache(handle, path, frame_range, sample_rate)`

原则：

- 结构数据只在 build 阶段进入
- 运行时不暴露底层求解器对象
- Python 只看见 handle、数组和轻量配置

---

### 10. 路线规划

#### 第一阶段：MVP 架构验证

- Blender 读取一条骨链
- 一键 build 成 C++ runtime scene
- C++ 保留 scene handle
- 每帧输入 root / animated target
- C++ step
- 返回骨骼变换
- Blender 同步 pose
- 支持 `reset / rebuild / destroy`

#### 第二阶段：组件系统与工作流

- 落地主组件列表与类型容器的双层结构
- 建立 registry
- 建立 `component_id` 主键机制
- 实现组件新建、删除、复制、重排
- 一键把骨链转成 XPBD 解算对象
- 自动碰撞体生成
- 本地调试可视化

#### 第三阶段：性能评估与能力上限

- 评估 IO 是否是瓶颈
- 评估 Blender pose 回写成本
- 评估 mesh cache 输出成本
- 引入更完整的碰撞过滤、空间哈希与缓存导出
- 扩展到粒子布料

---

### 11. 关键参考方向

- XPBD 最小实现
- PBD / XPBD 的距离约束、弯曲约束、shape matching
- Magica Cloth 2 的工作流体验
- Blender 侧 3D 视口调试与作者工具设计

---

### 12. 当前最优先事项

1. 保持 `nanobind + CMake + hocloth_native` 构建链稳定
2. 打通骨链 `build -> step -> pose` 闭环
3. 稳定 authoring 数据到 compiled 数据的导出层
4. 逐步把 C++ 占位 runtime 替换为正式 XPBD 求解器

---

### 13. 总结

HoCloth 当前的工程主线已经明确：

- **实时主线**：骨骼链物理
- **离线主线**：代理网格 / 布料缓存
- **架构主线**：Blender 做 authoring，C++ 做 runtime
- **实现主线**：先维持稳定接口和构建链，再逐步把占位 backend 替换成正式的自研 XPBD
