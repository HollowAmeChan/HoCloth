# HoCloth 工程大纲

## 1. 项目定位

HoCloth 的目标是把 Magica Cloth 2 风格的角色二级动力学系统带到 Blender 中，形成可实时预览、可离线烘焙、可导出缓存的布料与骨骼动力学插件。

当前主线已经正式切换为：

```text
Blender/Python authoring + MC2 Core 风格 C++ native backend
```

这不是继续写一个自研 XPBD MVP，也不是在三四个 C++ 文件里继续堆求解逻辑。新的方向是完整照搬 Magica Cloth 2 的 Core 架构思想，先把 C++ 侧后端系统建完整，再考虑 Blender 的逐帧数据交换与结果回写。

参考源码：

```text
_ReferenceProject/MagicaCloth2/Scripts/Core
```

核心规划文档：

```text
_docs/MC2_Core完整移植重构方案.md
```

## 2. 总体原则

1. MC2 Core 是物理后端的直接蓝本，不再只是“参考”。
2. C++ 侧优先完整移植系统架构，再追求单个功能的视口效果。
3. 现有 `_native` 代码只作为历史桥接参考，不再作为新架构中心。
4. Blender 侧负责 authoring、compiled data、runtime session、逐帧输入输出与回写。
5. Solver 内部不接收 Blender 对象，不依赖 Python dict，不把 UI 属性直接当运行时状态。
6. 任何直接移植或近似直译的代码块都必须注释 MC2 来源文件。
7. 不主动做算法再设计；只有语言适配、内存模型、Blender 接入和明确 bug 可以做必要调整。

## 3. 产品目标

### 3.1 BoneSpring

第一条贯通链。目标不是单独写一个弹簧骨，而是在新 MC2 风格后端里跑通 BoneSpring 所需的 team、transform、simulation、constraint、collider 数据流。

典型场景：

- 头发
- 尾巴
- 挂件链
- 裙摆骨
- 耳朵骨

### 3.2 ClothBone

目标还原 MC2 的骨驱动布料。它应复用同一套 manager 和 constraint pipeline，而不是另起一套 BoneSpring 变体。

关键能力：

- 多 root 骨集合
- 骨网络拓扑
- 距离 / 弯曲 / 形状保持约束
- 骨姿态批量回写
- 碰撞体、风场、固定点联动

### 3.3 MeshCloth

目标还原 MC2 的代理网格布料。

关键能力：

- render mesh / proxy mesh 编译
- reduction / selection / mapping
- simulation mesh 到 render mesh 映射
- 实时低模预览
- 高模缓存输出

## 4. Python / Blender 分层

```text
authoring/   用户编辑、选择提取、operator、handler、gizmo、工具
components/  Blender 属性组、组件容器、preset、组件注册
compile/     Blender 对象 -> compiled scene，拓扑校验，debug export
runtime/     native bridge、session 生命周期、逐帧输入、结果回写、缓存
```

Blender 侧的重点是把编辑态数据稳定编译成 native 可消费的数据，并在播放时执行：

```text
compiled scene -> build native session
frame inputs   -> native step
outputs        -> pose/mesh/cache apply
```

## 5. Native 后端目标架构

新的 `_native` 需要按 MC2 Core 模块拆分，而不是继续用少数 cpp 文件承载所有逻辑。

推荐结构：

```text
_native/
  include/hocloth/
    api/
    core/
    cloth/
    manager/
    simulation/
    virtual_mesh/
    utility/
  src/
    api/
    core/
    cloth/
      collider/
      constraints/
      wind/
    manager/
      cloth/
      render/
      simulation/
      team/
      transform/
      virtual_mesh/
    prebuild/
    reduction/
    utility/
      data/
      grid/
      math/
      mesh/
      native_collection/
      result_code/
      time/
    virtual_mesh/
      function/
    binding/
  tests/native/
  tools/
```

### 5.1 基础层

对齐 MC2 的 `Define / Interface / Utility`：

- result code
- math utility
- AABB / IntAABB
- bit flag
- chunk / range
- processing list
- native collection 等价容器
- time step 工具

### 5.2 Manager 层

必须建立，而不是跳过：

- `MagicaManager`
- `TimeManager`
- `TeamManager`
- `TransformManager`
- `SimulationManager`
- `ColliderManager`
- `WindManager`
- `ClothManager`
- `VirtualMeshManager`
- `RenderManager` 的数据边界

Blender runtime session 可以包住这些 manager，但不能替代这些 manager。

### 5.3 约束与模拟层

按 MC2 的约束系统逐个移植：

- `DistanceConstraint`
- `TriangleBendingConstraint`
- `TetherConstraint`
- `AngleConstraint`
- `MotionConstraint`
- `InertiaConstraint`
- `ColliderCollisionConstraint`
- `SelfCollisionConstraint`
- `SpringConstraint`

BoneSpring、ClothBone、MeshCloth 都应该最终落在同一套 simulation pipeline 上。

### 5.4 VirtualMesh / PreBuild 层

这是 ClothBone 和 MeshCloth 的核心前置系统：

- selection data
- virtual mesh primitive
- vertex attribute
- bone weight
- transform mapping
- mesh reduction
- proxy mesh
- input/output mapping
- serialization / debug dump

## 6. 数据边界

### 6.1 Authoring

Blender 中可编辑、可显示、可保存的组件和 UI 参数。

### 6.2 Compiled

稳定、扁平、可序列化、可 debug dump 的中间数据。它必须脱离 Blender 对象生命周期。

### 6.3 Runtime

native manager、buffer、constraint、particle、transform、collider、virtual mesh 的运行态。

每帧高频输入只包括：

- `dt`
- root / center / bone transform
- 外力、风、播放状态
- 少量 runtime override

结构变化统一 rebuild。

## 7. Native API 方向

```text
create_session() -> handle
destroy_session(handle)
build_scene(handle, compiled_scene)
reset_scene(handle)
set_frame_inputs(handle, frame_inputs)
step_scene(handle, dt)
get_bone_outputs(handle)
get_mesh_outputs(handle)
dump_debug_state(handle, path)
```

API 层只暴露 session 和 buffer 结果，不暴露内部 manager 指针。

## 8. 调试与验收

新路线必须从第一天建立可诊断性：

- compiled scene dump
- native build dump
- manager buffer dump
- constraint data dump
- per-step counters
- transform / collider / particle range dump
- MC2 源文件对应表

每个阶段都要能回答：

- 输入是否正确？
- native build 是否正确？
- manager range 是否正确？
- constraint buffer 是否正确？
- fixed step 是否正确？
- Blender 坐标转换是否正确？

## 9. 分阶段路线

### 阶段 A：架构落地

建立新 `_native` 目录结构、CMake 分组、空 runtime、最小 Python import。

### 阶段 B：基础设施移植

移植 `Define / Interface / Utility / Time / ResultCode / NativeCollection` 的 C++ 等价层。

### 阶段 C：Manager 骨架

建立 Team、Transform、Simulation、Collider、Wind、VirtualMesh、Cloth manager 生命周期。

### 阶段 D：BoneSpring 贯通

用新 manager 和 constraint pipeline 重做 BoneSpring。重点是行为对齐和可调试，不是继续修旧 solver。

### 阶段 E：ClothBone

接入骨网络拓扑、约束与骨姿态回写。

### 阶段 F：MeshCloth

接入 virtual mesh、proxy、mapping、cache 输出。

## 10. 当前优先事项

1. 停止继续扩展旧 `_native/src/hocloth_*.cpp` 作为物理核心。
2. 建立 `MC2_PORTING_MAP.md`，列出 MC2 Core 每个 `.cs` 文件的移植状态。
3. 先移植基础层和 manager 空壳。
4. 再让 BoneSpring 作为第一条穿透链跑通。
5. 保持 Blender 侧桥接能力，但不要让 Blender 侧补齐 solver 内部系统缺口。
## Blender / Native 数据交换契约

Python 侧已建立统一交换契约模块：

```text
runtime/exchange.py
_docs/Blender_Native数据交换契约.md
```

当前协议为 `hocloth.exchange v1`。Python debug/export 使用 envelope，native nanobind 入口已经可以识别 envelope，并保留旧 root dict 兼容解析。后续新增跨端字段应优先进入这个契约，而不是继续让 panel/operator/session 各自拼 dict。
